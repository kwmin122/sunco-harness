# Requirements

Generated by `/sunco:new`. Each requirement is verifiable — it can be checked by a test, an automated check, or a documented acceptance procedure. Requirements are written in user-behavior language, not implementation language.

**Format:** `REQ-NNN: [description]` — one behavior per requirement, no implementation details.

---

## How to Write Good Requirements

A well-formed requirement:
- Describes **observable behavior**, not implementation ("the system stores X" not "use a database")
- Has exactly **one thing to verify** — not compound behaviors joined with "and"
- Names the **actor** ("user", "system", "admin") explicitly
- Specifies the **outcome** clearly enough that a tester can write a pass/fail check without asking clarifying questions
- Avoids vague terms: "fast", "easy", "simple", "intuitive" — replace with measurable thresholds

**Quality checklist (before adding a requirement):**
- [ ] Can I write a failing test for this before implementing it?
- [ ] Is the actor explicit?
- [ ] Is the outcome observable without access to internal state?
- [ ] Is this a single behavior (not two behaviors joined with "and")?
- [ ] Have I named the priority (MUST / SHOULD / COULD / WON'T)?
- [ ] Is it free of implementation details?

## Category Taxonomy

Every requirement belongs to exactly one category. Use the category prefix in the ID for instant filtering.

| Code | Category | Description |
|------|----------|-------------|
| CORE | Core functionality | The primary user-facing behaviors that define the product |
| DATA | Data management | Storage, retrieval, migration, integrity, backup |
| UX | User experience | Interface behavior, feedback, accessibility, responsiveness |
| AUTH | Authentication & authorization | Login, permissions, roles, session management |
| PERF | Performance | Latency thresholds, throughput targets, resource limits |
| INFRA | Infrastructure | Build, deploy, monitoring, configuration, environment |
| INTG | Integration | Third-party APIs, webhooks, external service contracts |
| SEC | Security | Input validation, encryption, audit logging, threat mitigation |

**ID format with category:** `REQ-CORE-001`, `REQ-DATA-001`, `REQ-UX-001`, etc.

## MoSCoW Priority Scoring

| Priority | Label | Meaning |
|----------|-------|---------|
| 1 | MUST | Required for launch. Product is not viable without it. |
| 2 | SHOULD | High value, expected by users, but a workaround exists. |
| 3 | COULD | Nice to have. Included if time permits. |
| 4 | WON'T | Explicitly out of scope for this milestone. May be revisited. |

## Validation Workflow

When a requirement is implemented, the following steps must be completed before marking it `verified`:

1. **Unit test written** — test file path recorded in traceability matrix
2. **Integration test written** — end-to-end scenario covering the requirement
3. **Manual acceptance check** — tester confirms behavior against requirement description
4. **Edge cases documented** — at least two edge cases per requirement tested
5. **Regression guard** — requirement added to the regression suite so it cannot silently break

**Acceptance procedure for requirements without automated tests:**

```
Given: [precondition]
When: [actor performs action]
Then: [observable outcome]
Verified by: [person, date, method]
```
