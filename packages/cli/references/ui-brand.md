# UI/UX Design Guidelines for SUNCO Projects

How SUNCO detects existing design systems, selects component libraries, and enforces consistency standards during `/sunco:ui-phase` and `/sunco:ui-review`. Applied to all UI-facing phases.

---

## Design System Detection

Before generating any UI, SUNCO scans the project for an existing design system. Building on top of an existing system is always preferred over introducing a new one.

### Detection order

1. **Explicit declaration** — Check `.planning/UI-SPEC.md` for `design_system:` field
2. **Package scan** — Check `package.json` dependencies for known UI libraries
3. **Class pattern scan** — Check source files for recognizable class naming patterns
4. **Import pattern scan** — Check for known import paths

### Known design systems and detection signals

| Design system | Detection signal | Component approach |
|--------------|------------------|--------------------|
| Tailwind CSS | `"tailwindcss"` in deps + `tailwind.config.*` file | Utility classes, no component library |
| shadcn/ui | `"@radix-ui/*"` in deps + `components/ui/` directory | Radix primitives + styled with Tailwind |
| Chakra UI | `"@chakra-ui/react"` in deps | Component library with theme tokens |
| Mantine | `"@mantine/core"` in deps | Component library, CSS modules |
| Ant Design | `"antd"` in deps | Enterprise component library |
| Material UI | `"@mui/material"` in deps | Material Design implementation |
| Custom system | `design-tokens.*` file or `src/design-system/` directory | Follow custom system rules |

### When no system is detected

If no design system is found, SUNCO recommends Tailwind CSS + shadcn/ui for new web projects. The user confirms via `/sunco:discuss` before UI-SPEC.md is written.

Never mix two design systems in the same project. If one is detected, use it exclusively.

## Component Library Selection

Given a design system, choose components in this priority order:

1. **Use what exists** — If the project already has a component for this purpose, use it. Check `components/` or `src/components/` first.

2. **Extend an existing component** — If a similar component exists but lacks a needed prop, extend it (not duplicate it).

3. **Compose from primitives** — If using shadcn/ui or Radix, compose from their primitives rather than writing from scratch.

4. **Create a new component** — Last resort. New components must be added to the project's component library, not one-offs in page files.

### Component naming conventions

- PascalCase for component files: `UserCard.tsx`, not `user-card.tsx`
- Props interface named `[Component]Props`: `UserCardProps`
- Export the props type alongside the component for composability
- Keep component files under 150 lines — extract sub-components if larger

## Accessibility Standards

All UI components generated by SUNCO must meet WCAG 2.1 AA standards. These are non-negotiable defaults.

### Required accessibility checks

| Check | Standard | How to verify |
|-------|----------|---------------|
| Color contrast (text) | 4.5:1 ratio | Use browser devtools contrast checker |
| Color contrast (large text, 18px+) | 3:1 ratio | Same |
| Interactive element focus visibility | Visible focus ring | Tab through all interactive elements |
| Images have alt text | Non-decorative images | `<img alt="description">` |
| Form inputs have labels | Programmatic association | `<label htmlFor>` or `aria-label` |
| Buttons have accessible names | Non-empty accessible name | Button text or `aria-label` |
| Keyboard navigation | All functionality reachable | Tab, Enter, Escape, Arrow keys |
| Screen reader headings | Logical heading hierarchy | h1 → h2 → h3, no skipped levels |

### Accessibility anti-patterns

```tsx
// Bad: click handler on non-interactive element
<div onClick={handleClick}>Click me</div>

// Good: use semantic interactive element
<button onClick={handleClick}>Click me</button>

// Bad: no label on input
<input type="email" placeholder="Email" />

// Good: labeled input
<label htmlFor="email">Email</label>
<input id="email" type="email" placeholder="your@email.com" />

// Bad: icon button with no accessible name
<button><SearchIcon /></button>

// Good: icon button with aria-label
<button aria-label="Search"><SearchIcon /></button>
```

## Mobile-First Patterns

All UI phases default to mobile-first. Desktop styles are added via responsive breakpoints on top of mobile base styles.

### Breakpoint strategy (Tailwind)

```
Base (mobile):    no prefix     (0px+)
Small:            sm:           (640px+)
Medium:           md:           (768px+)
Large:            lg:           (1024px+)
Extra large:      xl:           (1280px+)
```

### Mobile-first component pattern

```tsx
// Mobile-first: base is mobile, larger screens add complexity
<nav className="
  flex flex-col gap-2       // mobile: vertical stack
  sm:flex-row sm:gap-4      // tablet+: horizontal row
  lg:gap-8                  // desktop: more spacing
">
```

**Never start with desktop and shrink down.** Media query `max-width` patterns are mobile-last, not mobile-first.

### Touch targets

Interactive elements must be at least 44x44px on mobile:

```tsx
// Minimum touch target
<button className="min-h-[44px] min-w-[44px] p-2">
  <Icon />
</button>
```

## SUNCO Terminal UI Standards (CLI output)

For SUNCO itself (CLI output rendered via Ink), design standards differ from web UI.

### Color system for terminal output

| Use | Color | Ink component |
|-----|-------|---------------|
| Success | Green | `<Text color="green">` |
| Error | Red | `<Text color="red">` |
| Warning | Yellow | `<Text color="yellow">` |
| Info | Cyan | `<Text color="cyan">` |
| Muted/secondary | Gray | `<Text dimColor>` |
| Emphasis | Bold | `<Text bold>` |
| Path/code | No color, dim | `<Text dimColor>` |

### Terminal output hierarchy

```
Primary action result     ← Full brightness, bold if success/error
Progress indicator        ← Dim while in progress, full on complete
Secondary info            ← dimColor, never color
Error with context        ← Red header, white context lines
Recommendation/tip        ← Cyan prefix "→", normal text body
```

### Output width constraints

Wrap text at 80 characters for compatibility with default terminal widths. Never assume a wide terminal. Use `<Box flexWrap="wrap">` in Ink components.

### Loading and progress

Always show a spinner for operations > 500ms. Show a progress bar for operations with a known total:

```tsx
// Operation duration unknown → spinner
<Spinner type="dots" />
<Text> Scanning project...</Text>

// Operation with known total → progress bar
<Text>Verifying [{done}/{total}] layers</Text>
```

## UI Review Checklist (6 Axes)

Applied by `/sunco:ui-review` to score UI implementations.

### Axis 1: Visual Hierarchy (0-10)
- Clear primary action visible without scrolling
- Size and weight differentiate content importance
- Whitespace creates breathing room between sections

### Axis 2: Consistency (0-10)
- Same interaction, same visual treatment (buttons look like buttons)
- Consistent spacing using design system tokens, not arbitrary pixel values
- Color used consistently (success is always green, not sometimes blue)

### Axis 3: Responsiveness (0-10)
- Layout works on 375px (mobile), 768px (tablet), 1280px (desktop)
- No horizontal overflow at any breakpoint
- Touch targets meet 44px minimum on mobile

### Axis 4: Accessibility (0-10)
- WCAG 2.1 AA contrast ratios met
- Keyboard navigation complete
- Screen reader labels present on all interactive elements

### Axis 5: Performance (0-10)
- No layout shift on load (reserved space for images/async content)
- No render-blocking assets
- List rendering uses virtualization when list length > 100 items

### Axis 6: Copy and Tone (0-10)
- Error messages tell user what to do, not just what went wrong
- Button labels are verbs ("Save", not "OK")
- Loading states have informative messages ("Checking credentials...", not "Loading...")

**Minimum score to ship:** 7/10 on each axis. Any axis below 5 is a blocker.
