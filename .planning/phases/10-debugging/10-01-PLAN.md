    ```typescript
    // Failure classification for debug skill (D-01)
    export type FailureType = 'context_shortage' | 'direction_error' | 'structural_conflict';

    // Individual error items
    export interface DiagnoseError {
      type: 'test_failure' | 'type_error' | 'lint_error';
      file: string;
      line: number | null;
      message: string;
      code?: string;  // TS error code or ESLint rule
      stack?: string;  // stack trace for test failures
    }

    // Full diagnose result (D-08)
    export interface DiagnoseResult {
      test_failures: DiagnoseError[];
      type_errors: DiagnoseError[];
      lint_errors: DiagnoseError[];
      total_errors: number;
      raw_output: {
        test?: string;
        tsc?: string;
        lint?: string;
      };
    }

    // Debug analysis result (D-03)
    export interface DebugAnalysis {
      failure_type: FailureType;
      root_cause: string;
      affected_files: { file: string; line?: number; reason: string }[];
      fix_suggestions: { action: string; file?: string; priority: 'high' | 'medium' | 'low' }[];
      confidence: number;  // 0-100
    }

    // Forensics report (D-10)
    export interface ForensicsReport {
      timeline: { timestamp: string; event: string; source: string }[];
      divergence_point: string;
      root_cause_hypothesis: string;
      affected_plans: string[];
      prevention_recommendations: string[];
    }
    ```

    **diagnose.skill.ts** -- Deterministic log analysis skill (per D-05, D-06, D-07, D-08):

    - `defineSkill({ id: 'workflow.diagnose', command: 'diagnose', kind: 'deterministic', stage: 'stable', category: 'workflow', routing: 'routable' })`
    - Options: `--test-only` (only run tests), `--type-only` (only run tsc), `--lint-only` (only run eslint)
    - Execute flow:
      1. `ctx.ui.entry()` with "Diagnose: Analyzing build/test output..."
      2. `ctx.ui.progress()` with total=3 (test, tsc, lint)
      3. **Test step** (unless --type-only or --lint-only): spawn `npx vitest run --reporter=json` via `promisify(execFile)` with 120s timeout. Parse JSON output to extract test failures. Pattern: `.testResults[].assertionResults[].status === 'failed'` gives `fullName`, `failureMessages`, `ancestorTitles`. Extract file from `.testResults[].name`.
      4. **TSC step** (unless --test-only or --lint-only): spawn `npx tsc --noEmit --pretty false` via `execFile`. Parse each line matching `^(.+)\((\d+),(\d+)\): error (TS\d+): (.+)$` for file, line, col, code, message.
      5. **Lint step** (unless --test-only or --type-only): spawn `npx eslint . --format json` via `execFile`. Parse JSON output to extract errors: `.messages[].ruleId`, `.messages[].line`, `.messages[].message`, `.filePath`.
      6. Each step wrapped in try/catch -- failure of one does NOT block others (same Swiss cheese pattern as verify).
      7. Build `DiagnoseResult`, count total errors.
      8. Store in `ctx.state.set('diagnose.lastResult', result)`.
      9. `ctx.ui.result()` with success = (total_errors === 0), details listing counts per category.
      10. Return `{ success, summary, data: result }`.

    Export 3 parser functions for testability: `parseTestOutput(json: string)`, `parseTypeErrors(output: string)`, `parseLintErrors(json: string)` -- each returns `DiagnoseError[]`.

    **Tests** -- Unit tests for the 3 parsers using realistic vitest/tsc/eslint output samples.

    **debug-analyze.ts** -- Prompt builder for `sunco debug` (per D-01, D-02, D-03):

    `buildDebugAnalyzePrompt(params: { gitLog: string; testOutput: string; buildOutput: string; stateSnapshot: string; recentErrors: string })` returning a string prompt.

    The prompt instructs the agent to:
    1. Analyze the failure context (git history, test output, build logs, .sun/ state).
    2. Classify the failure as one of: `context_shortage` (agent ran out of context or had incomplete info), `direction_error` (agent took wrong approach), `structural_conflict` (codebase architecture prevents the change).
    3. Identify root cause with specific files and lines.
    4. Suggest actionable fixes with priority (high/medium/low).
    5. Output structured JSON matching `DebugAnalysis` interface.
    6. MAX_CHARS = 50_000 truncation on all input strings (same pattern as verify-security.ts).

    **forensics-postmortem.ts** -- Prompt builder for `sunco forensics` (per D-09, D-10):

    `buildForensicsPostmortemPrompt(params: { gitHistory: string; planFiles: string; summaryFiles: string; verificationReports: string; stateHistory: string })` returning a string prompt.

    The prompt instructs the agent to:
    1. Reconstruct a timeline of events from git commits, plan execution, and verification.
    2. Identify where the workflow diverged from the plan.
    3. Hypothesize the root cause of the failure.
    4. List affected plans.
    5. Recommend prevention measures.
    6. Output structured JSON matching `ForensicsReport` interface.
    7. MAX_CHARS = 50_000 truncation per input section.

    Both follow the exact pattern of verify-security.ts: function export, MAX_CHARS const, truncation with notice, template literal prompt with sections, JSON output format block at the end with "Only output the JSON. No explanation before or after."
