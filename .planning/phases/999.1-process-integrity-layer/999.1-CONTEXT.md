# Backlog 999.1 — SUNCO Process Integrity Layer

- **Spec alias**: backlog (outside active v1.4 phase sequence 35-51)
- **Target absorption**: Phase 52 (v1.4 tail) or M5 Rollout Hardening
- **Status**: captured (2026-04-18) — not to interrupt Phase 40-41 flow
- **Source**: Session-learned patterns from v1.4 Phase 35-39 implementation that deserve to become first-class SUNCO protocol rather than per-session tribal knowledge.

## Why

SUNCO's current value is "helping plan well." Process Integrity Layer elevates it to "validating its own state before acting." Four patterns repeatedly surfaced and prevented bugs during v1.4 M1/M2 execution — they should be structural, not recalled from memory each session.

## Four components

### 1. Planning Source Freshness Gate

**Problem observed**: Session discovered STATE.md claimed v1.2 executing while ROADMAP.md was on v1.3 Phase 34. Single-file belief is unsafe — artifacts drift independently.

**Rule**: Before mutating any planning artifact, compare mtime + last-touching-commit across:
- `PROJECT.md`
- `REQUIREMENTS.md`
- `ROADMAP.md`
- `STATE.md`
- `.planning/phases/*/CONTEXT.md`

Emit a "Planning Source Freshness" report with per-file authority, last commit, and mismatch summary. Block mutation if mismatches exceed threshold without explicit override.

**Insertion points**:
- `/sunco:context`

- `/sunco:milestone`
- `/sunco:pivot`
- `/sunco:reinforce`

**Optional new surface**: `/sunco:freshness` or `/sunco:planning-audit` as stand-alone entry.

### 2. Judge Lane / Gate Delegation Rule

**Problem observed**: Mid-session judges re-adjudicated decisions already bounded by closed gates — verification loops ran too long.

**Rule**: When a gate approves a bounded implementation space, the implementer may choose among judge-proposed options (or propose new ones) if:
1. The choice preserves the gate invariant
2. Rationale is recorded in the commit body
3. The change is a local additive within delegated scope

**Re-review required** only when:
1. Scope expands beyond the delegated bound
2. A RED condition appears
3. Fresh evidence contradicts the gate premise
4. Destructive or third-party-visible action is introduced
5. Global API / public contract change
6. New UX surface (flag, env var, config option)

**Line count is NOT a threshold.** Scope membership decides.

### 3. Session-Derived Invariant (SDI) Registry

**Problem observed**: Phase 39/M2.2 produced SDI-1 ("SUNCO never writes .impeccable.md"). It isn't a spec-locked R1-R6 rule — it emerged from implementation. Treating session-derived invariants identically to spec rules risks confusing future contributors.

**Rule**: Maintain a separate registry for SDIs with distinct provenance.

**Candidate artifact**: `.planning/INVARIANTS.md`

**Do NOT create until Process Integrity Layer design is accepted (Phase 52 or M5).** Schema for SDI vs spec-rule separation, registry structure, lifecycle rules (active / retired / superseded) must be decided in PIL scope before materializing the file. Until then, SDIs live in session memory + this 999.1-CONTEXT.md. Creating the file prematurely would force Phase 52 to start with refactor work rather than greenfield design.

### 4. Stable Checkpoint / Push Boundary

**Problem observed**: Every v1.4 phase commit through f50ebdb required explicit user push approval. Pattern worked — stable work pushed, risky next phase always had a clean rollback target.

**Rule**: Before entering a risky phase (external vendoring, workflow refactor, install pipeline change, migration, destructive action, cross-domain verify gate change):

1. If current phase is complete AND working tree is clean:
2. Commit local checkpoint (already done for completed phase)
3. Present status: HEAD, working tree, origin ahead/behind
4. Ask user before push (never auto-push)
5. Do not push tags unless explicitly requested
6. Record rollback command in emitted report

**Example output**:
```
Risky phase ahead: Phase 40 ui-phase-web behavior.

Stable checkpoint available:
- HEAD: f50ebdb
- working tree: clean
- origin/main: behind by 0

Recommendation:
Push current stable checkpoint before entering Phase 40.
Command requires explicit approval:
  git push origin main

Rollback if Phase 40 fails:
  git reset --hard f50ebdb
```
