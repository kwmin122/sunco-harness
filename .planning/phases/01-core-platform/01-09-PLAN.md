```typescript
export interface Recommendation {
  skillId: string; command: string; reason: string;
  priority: number; recommended: boolean;
}

export interface RecommendationRule {
  id: string; description: string;
  match: (state: RecommendationState) => boolean;
  recommend: (state: RecommendationState) => Recommendation[];
}

export interface RecommendationState {
  lastSkillId: string | null;
  lastResult: SkillResult | null;
  currentPhase: string | null;
  hasUncommittedChanges: boolean;
  availableSkills: string[];
}

export interface RecommenderApi {
  getRecommendations(state: RecommendationState): Recommendation[];
}
```

  <behavior>
    **engine.ts:**
    - createRecommender(rules) returns RecommenderApi
    - getRecommendations(state) iterates rules, collects matches, sorts by priority, limits to 4 max
    - getRecommendations ensures exactly one recommendation has recommended=true (highest priority)
    - getRecommendations with no matching rules returns fallback recommendation: "sunco status"
    - getRecommendations completes in < 1ms for 50 rules (benchmark test)

    **rules.ts - Core workflow routing (REC-03):**
    - After execute success -> recommend verify (priority 1, recommended), then ship (priority 3)
    - After verify success -> recommend ship (priority 1, recommended)
    - After verify failure -> recommend debug (priority 1, recommended), then execute (priority 2)
    - After plan success -> recommend execute (priority 1, recommended)
    - After discuss success -> recommend plan (priority 1, recommended), then research (priority 2)
    - After init success -> recommend lint (priority 1, recommended), then health (priority 2)
    - After ship success -> recommend milestone audit (priority 2)

    **rules.ts - State-based context rules:**
    - If hasUncommittedChanges and lastSkill was execute -> recommend ship (higher priority)
    - If currentPhase is null -> recommend new or scan (priority 1)
    - If no lastSkillId (fresh session) -> recommend status (priority 1, recommended)

    **rules.ts - Fallback rules:**
    - Generic fallback: recommend status (priority 99)

    Total: aim for 25-30 rules covering all major transitions.
  </behavior>
  <implementation>
    **engine.ts:**
    - RecommenderEngine class implementing RecommenderApi
    - Constructor accepts RecommendationRule[]
    - getRecommendations: iterate rules, for each rule where match(state) returns true, collect recommend(state) results. Flatten, sort by priority ascending. Slice to max 4. Set recommended=true on first item only (clear others). If empty, return fallback.

    **rules.ts:**
    - Export const RECOMMENDATION_RULES: RecommendationRule[]
    - Each rule: { id: 'after-execute-success', description: '...', match: (s) => s.lastSkillId === 'workflow.execute' && s.lastResult?.success === true, recommend: (s) => [{ skillId: 'workflow.verify', command: 'verify', reason: 'Verify agent output', priority: 1, recommended: false }] }
    - The recommended flag is set false in rules -- the engine sets the top priority one to true
    - Group rules by category for readability: workflow-transition, state-context, fallback

    Rules to implement (25+):
    1-7: Workflow chain transitions (discuss->plan->execute->verify->ship)
    8-12: Harness skill transitions (init->lint->health->guard)
    13-16: Session state rules (fresh session, no phase, uncommitted changes)
    17-20: Error recovery rules (verify fail, execute fail, any fail)
    21-24: Milestone rules (after ship, after milestone)
    25-30: Context-aware rules (available skills filter, phase-specific suggestions)

    **index.ts:** Export createRecommender (factory that uses RECOMMENDATION_RULES), export RECOMMENDATION_RULES for testing
  </implementation>
</feature>
